## 3. 关键实现说明

### 3.1 共享请求模型
- 新增 `AppExecutionRequest` / `AppExecutionAction`：
  - 包含 `requestID`、`action`、`directoryPath`、文件创建参数等。

### 3.2 转发实现（已确认）
- 主通道：`DistributedNotificationCenter`
  - 通知名：`com.superright.app.execution.request`
  - 负载：JSON 字符串（`payload`）
- 防丢机制：App Group 请求队列（新增）
  - 目录：`~/Library/Group Containers/group.com.superright.shared/app_execution_requests`
  - Finder Extension 在转发前先落盘 `requestID.json`
  - 主 App 在启动时和收到通知时都会消费队列
- 自动拉起：Extension 在转发前检查主 App 运行态；若未运行，用 `NSWorkspace` 按 bundle id 发起启动请求。

### 3.3 Finder Extension 侧
- `FinderSync.handleMenuItem` 中：
  - `createFolder/createFile` 先走 `forwardToMainAppIfNeeded(...)`
  - 先写入 App Group 请求队列，再发分布式通知唤醒主 App
  - 转发成功后记 `Action forwarded to main app` 并返回
  - 其他动作继续本地 `ActionDispatcher`

### 3.4 主 App 侧
- 新增 `AppExecutionRequestHandler` 常驻监听通知：
  - 启动即安装监听器并执行一次队列消费（`source=startup`）
  - 收到通知后再次消费队列（`source=notification`）
  - 若通知内附带 payload，则写回队列并再消费一次（`source=notification-payload`）
  - 执行目录/文件创建并记录成功/失败
- 使用 `AppDelegate` 托管监听器生命周期（`applicationDidFinishLaunching` 安装），避免 SwiftUI `App` 结构体重建导致监听丢失
- 日志写入：
  - `NSLog`
  - `~/Library/Group Containers/group.com.superright.shared/app_execution_debug.log`
  - 主 App 侧补齐链路日志：
    - `app request received`
    - `requestID`
    - `action type`
    - `target directory`
    - `whether app is already running`
    - `main app handler entered`
    - `createFolder started`
    - `createFolder success/failed`
    - `failure reason`

### 3.5 Debug 权限配置
- 新增 `右键增强.debug.entitlements`（不含 `app-sandbox`）
- 主 App Debug 改为使用该 entitlements
- 主 App Release 维持原 `右键增强.entitlements`（含 sandbox）

## 5. 主 App 运行要求结论

- 支持“主 App 未运行时请求不丢失”：
  - Extension 会尝试自动拉起主 App。
  - 请求先落盘队列，主 App 启动后会消费队列，即使首条通知丢失也可补偿执行。
- 仍存在平台级不确定性：
  - 若系统阻止自动拉起（签名/权限/用户策略），请求会留在队列，需主 App 手动启动后才执行。
  - 因此当前结论不是“100% 无条件自动执行”，而是“自动拉起 + 队列补偿”。

## 6. 当前剩余限制

- 当前 Finder 扩展监听目录仍是桌面（`/Users/maple/Desktop`），不是全局目录。
- Debug 与 Release 权限模型不同（Debug 联调放开，Release 保守沙盒）。
- P1 中“终端/iTerm/VSCode/复制路径”仍在 Extension 内执行，需按场景逐项回归验证。
